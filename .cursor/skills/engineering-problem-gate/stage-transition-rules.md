# Engineering Problem Gate 阶段切换规则

本文件用于把 `AI Delivery Loop` 里的关键判断点收紧成可执行规则。  
目标不是补更多理念，而是明确：

- 什么时候继续追问
- 什么时候进入分解
- 什么时候进入闭环
- 什么时候必须人审

## 一句话定义

这套规则的作用，是控制任务在不同阶段之间的切换，避免：

- 问题没看透就开始拆
- 还没拆清就启动 AutoResearch
- AI 输出还没收敛就假装可以交付

## 四个判断点

### 1. 继续追问

适用阶段：

- `See Through`

### 触发条件

只要出现下面任意一种情况，就继续追问，不进入下一阶段：

- 问题对象不明确
- 重构/设计/优化的对象不明确
- 当前痛点说不清
- 成功标准不可检验
- 已知约束过少
- 未知项没有显式写出
- 用户只给方向，没有给任务边界

### 继续追问的最小目标

至少追到下面 4 项中的大部分清楚：

1. 具体对象是什么
2. 为什么要做
3. 主要做哪一层
4. 什么结果算成功

### 继续追问时禁止做的事

- 不直接写技术方案
- 不直接写实现计划
- 不直接启动 AutoResearch
- 不用“我大概理解了”替代边界澄清

## 2. 进入分解

适用阶段：

- 从 `See Through` 进入 `Structure`

### 放行条件

只有同时满足下面条件，才进入分解：

1. 问题对象明确
2. 当前痛点明确
3. 重构/设计/处理范围明确
4. 成功标准具备基本可检验性
5. 已知约束至少有 1 到 2 条真实约束
6. 未知项已被显式列出

### 进入分解后的目标

不是直接开始实现，而是把任务拆成：

- 更小的子问题
- 明确输入 / 输出
- 可判断是否继续往下推进

### 不应误判为“可以进入分解”的情况

- 只是知道一个算法名或模块名
- 只是知道“要重构”
- 只是知道“现在不好用”
- 成功标准还是空话

## 3. 进入闭环

适用阶段：

- 从 `Structure` 进入 `Close the Loop`

### 什么叫“进入闭环”

不是开始大规模实现，而是：

- 某个子问题已经足够小
- 有局部目标
- 有输入 / 输出
- 有局部验收标准
- 可以通过多轮迭代逐步收敛

### 放行条件

只有同时满足下面条件，才可进入闭环：

1. 子问题只解决一个核心目标
2. 输入明确
3. 输出明确
4. 完成标准明确
5. 依赖数量可控
6. 目标不会在每轮迭代里频繁变化

注意：

- 进入闭环只表示这个问题已经适合做局部收敛
- 这并不自动等于可以启动 `AutoResearch`

### 适合进入闭环的问题类型

- 某个接口标准是否应统一
- 某个参数组如何分层
- 某个局部适配层如何抽象
- 某个局部代码生成规则如何稳定
- 某个标准排障流程如何固化

### 不应进入闭环的问题类型

- 大型系统总体架构
- 多角色共同拍板的问题
- 目标仍然频繁变化的问题
- 成功标准还没统一的问题
- 明显需要管理决策而不是技术迭代的问题

### AutoResearch 触发条件

只有在下面条件同时满足时，才建议把闭环问题升级为 `AutoResearch`：

1. 子问题已经进入闭环空间
2. 有明确的修改对象
3. 有明确的不可修改边界
4. 改动后可以运行真实实验
5. 有固定评估指标
6. 有保留 / 丢弃 / 回退规则
7. 有结果记录方式

换句话说：

- 所有 `AutoResearch` 都属于闭环
- 但不是所有闭环都应该变成 `AutoResearch`

### 适合升级为 AutoResearch 的问题

- 某个局部算法或参数面的实验筛选
- 某个局部架构候选在固定指标下的比较
- 某个模块在真实运行结果下的优化
- 某个局部生成规则在真实评测里的筛选

### 不应升级为 AutoResearch 的问题

- 总体架构拍板
- 多角色共同决策的问题
- 尚未定义固定评估指标的问题
- 只能靠主观讨论判断优劣的问题
- 还在频繁改目标的问题

## 4. 人审验收

适用阶段：

- 从 `Close the Loop` 进入 `Accept and Deliver`

### 必须人审的节点

下面这些节点不应省略人工判断：

1. 问题本质是否成立
2. 分解结构是否合理
3. 子问题是否真的进入闭环空间
4. 某个闭环问题是否真的满足 `AutoResearch` 启动条件
5. 候选方案是否满足关键约束
6. 输出是否达到交付标准

### 人审关注的核心问题

- 这个结果是否回答了原始问题
- 有没有违背不可绕过的约束
- 有没有过度设计
- 有没有把局部最优误当成整体正确
- 是否真的可以交付，而不是“AI 看起来像完成了”

### 人审后的可能结论

人审不是只有“通过”一种结果，至少有四种：

1. `通过，可交付`
2. `部分通过，继续局部迭代`
3. `退回到分解阶段`
4. `退回到问题澄清阶段`

## 推荐的阶段切换顺序

推荐链路如下：

1. `继续追问`
   - 问题没看透时停在这里
2. `进入分解`
   - 问题已清楚，但还没拆开
3. `进入闭环`
   - 子问题已经足够小
4. `判断是否启动 AutoResearch`
   - 只有满足实验条件才升级
5. `人审验收`
   - 结果开始接近可交付

## 常见错误切换

### 错误 1：跳过继续追问

现象：

- 只知道一个方向，就开始拆架构

后果：

- 拆出来的是伪结构，不是真问题结构

### 错误 2：跳过分解直接闭环

现象：

- 一上来就启动 AutoResearch

后果：

- AI 在错误边界上不断迭代

### 错误 3：把所有闭环都当成 AutoResearch

现象：

- 只要问题拆小了，就默认启动 `AutoResearch`

后果：

- 把普通局部迭代误升级成自治实验
- 在没有固定评估和回退机制时盲目循环

### 错误 4：跳过人审直接交付

现象：

- AI 输出看起来完整，就直接当答案

后果：

- 容易出现“形式完整、实质跑偏”

## 最小可执行规则

如果只保留最小骨架，可以记住这 5 句：

1. 问题没看透，就继续追问
2. 问题看透了，先分解，不先实现
3. 子问题没闭环，不进入实验循环
4. 没有固定评估和回退机制，不启动 `AutoResearch`
5. 没有人审，不算真正交付

## 一句话总结

这套阶段切换规则的核心，不是让流程更复杂，而是让每一步都只做它该做的事。
