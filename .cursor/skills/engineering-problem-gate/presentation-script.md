# Engineering Problem Gate 宣讲稿

## 一页式 5 分钟介绍

### 标题

`Engineering Problem Gate：让 Cursor 先收敛问题，再进入执行`

### 开场

大家在用 Cursor 的时候，最常见的问题不是“不会问”，而是“问题本身还没想清楚，就已经开始让 AI 给方案、写代码、做实现了”。

这样带来的结果通常是：

- AI 输出很多，但不贴项目实际
- 方案看起来完整，落地时发现边界没想清楚
- 不同角色各自和 AI 对话，但上下文不一致
- 工程师容易反复试错，效率并不高

### 我们做这个 Skill 的目的

`engineering-problem-gate` 不是为了多加一道流程，而是为了把 AI 使用前的关键动作标准化：

1. 先补项目背景和资源
2. 先把问题定义清楚
3. 先把任务拆到 AI 易执行的颗粒度
4. 再决定是否进入方案、实现，或者沉淀为 skill / harness

### 它解决的核心问题

它主要解决三类问题：

- 问题没定义清楚就开做
- 项目背景和资源没有先给 AI
- 任务颗粒太大，AI 很难稳定执行

### 它适合谁

适合公司内各类角色：

- 数字工程师
- 模拟工程师
- Layout 工程师
- 软件工程师
- 算法工程师
- 测试工程师
- 市场产品经理
- 现场支持工程师

### 它怎么组成

这套内容现在由四部分组成：

- `SKILL.md`：主门禁流程
- `project-intake.md`：项目启动前背景采集
- `role-fields.md`：不同角色扩展字段
- `decomposition-patterns.md`：第一性原理与结构化分解

### 最简单的使用方式

在开始做方案、代码、skill、harness 之前，先对 Cursor 说一句：

`请先用 engineering-problem-gate 帮我梳理这个任务，不要直接进入方案或实现。`

### 带来的价值

- 减少 AI 使用中的盲目推进
- 让输出更贴近真实项目
- 提高首次输出质量
- 帮助团队逐步沉淀更合理的 skill / harness

### 结束语

可以把它理解成一个统一入口：  
不是限制大家用 AI，而是让 AI 更早接触正确的问题、正确的背景和正确的任务颗粒度。

## 1 分钟极简版

我们新增了一个企业内 Cursor Skill，叫 `engineering-problem-gate`。  
它的作用是让大家在用 AI 做方案、写代码、造 skill、造 harness 之前，先把项目背景、问题定义和任务分解做清楚，避免 AI 直接跑偏。  
它适用于多角色协作项目，也适合模糊需求和新任务进入时使用。  
最简单的用法就是先对 Cursor 说：  
`请先用 engineering-problem-gate 帮我梳理这个任务，不要直接进入方案或实现。`

## 常见提问答法

### Q1：这是不是会让流程变重？

不是。它的目的是减少返工和跑偏。  
如果一个问题本来就很清楚，可以很快通过；真正需要它的是那些模糊任务、跨角色任务和项目启动场景。

### Q2：为什么不直接让 AI 自己判断？

因为 AI 是否能给出高质量结果，很大程度取决于输入的背景、边界和颗粒度。  
这套 skill 的作用就是把这些关键前提补齐。

### Q3：它和 harness、skill 有什么关系？

它本身就是一个 skill，但它还负责判断哪些模式后续值得沉淀成正式的 `skill` 或 `harness`。  
可以理解成“沉淀之前的统一入口”。
