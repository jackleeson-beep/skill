# Engineering Problem Gate v0.2 迭代草案

## 迭代目标

`v0.1` 已经具备完整框架，但更偏“方法完整”。  
`v0.2` 的目标不是继续加内容，而是让这套 skill 在真实项目里**更容易被工程师持续使用**。

本轮只解决三个核心问题：

1. 字段太多，第一次使用门槛偏高
2. “什么时候算通过门禁”还不够明确
3. 缺少一个更轻量的入口流程，容易让人觉得太重

## v0.2 的设计原则

1. **先降低首次使用门槛**
   - 先让工程师愿意用，再逐步补齐完整流程
2. **把放行规则说清楚**
   - 减少“到底现在该继续问还是该往下做”的模糊地带
3. **完整版保留，新增轻量版**
   - 不推翻 `v0.1`
   - 只增加更容易上手的使用路径
4. **继续服务真实项目，而不是只服务文档完整性**
   - 每个新增规则都要能回答：它是否真的减少了盲目推进？

## v0.1 当前主要问题

### 问题 1：信息密度高

`v0.1` 对完整流程定义得很充分，但对第一次接触的工程师来说，可能会觉得：

- 字段多
- 步骤多
- 很像完整方法论，而不是一个容易开口使用的工具

### 问题 2：门禁结论还不够“硬”

虽然已经有“通过 / 不通过”框架，但实操时还缺少更明确的最低通过标准。  
比如：

- 成功标准写成什么程度才算可检验
- 未知项列多少才算足够
- 什么时候可以直接进入结构化分解

### 问题 3：缺少轻量路径

不是所有任务都适合一上来走完整版。  
例如：

- 一个模糊但不复杂的问题
- 一个局部任务
- 一个还没正式进入项目立项的需求

这类情况更适合先走轻量版，再决定是否升级到完整版。

## v0.2 核心改动方向

### 方向 1：定义最小必填字段

不管什么角色、什么任务，第一次放行前至少先补齐下面 5 项：

1. `任务目标`
2. `当前现状`
3. `已知约束`
4. `成功标准`
5. `未知项`

这是 `v0.2` 的**最小门禁骨架**。

### 为什么是这 5 项

- 没有 `任务目标`，不知道在解决什么
- 没有 `当前现状`，不知道问题发生在什么上下文
- 没有 `已知约束`，方案会脱离现实
- 没有 `成功标准`，无法判断是否完成
- 没有 `未知项`，AI 容易偷偷脑补

## 建议的字段分层

### A. 最小必填字段

所有任务第一次放行前必须有：

- `任务目标`
- `当前现状`
- `已知约束`
- `成功标准`
- `未知项`

### B. 建议补充字段

当任务开始变复杂时，建议补：

- `影响范围`
- `当前阶段`
- `下一步允许动作`

### C. 角色扩展字段

只有在任务进入具体角色语境时再补。  
不要一开始就强迫所有人填满角色字段。

## v0.2 门禁通过条件

本轮建议把门禁通过条件收紧成下面这 6 条。

只有同时满足，才算**通过门禁**：

1. `任务目标` 不是空话  
   - 不能只是“优化一下”“更智能一点”“做得更好”

2. `当前现状` 能说清问题发生在哪里  
   - 至少要知道场景、对象或当前卡点

3. `已知约束` 至少有一条真实约束  
   - 时间、资源、技术、流程、客户、质量要求都算

4. `成功标准` 可检验  
   - 至少能回答“什么结果算完成”

5. `未知项` 被显式写出  
   - 不能假装没有未知

6. 下一步动作不越级  
   - 问题没清楚时不能直接进入实现

## v0.2 不通过条件

只要出现下面任意一种情况，就默认**不通过**：

- 任务目标过于空泛
- 成功标准无法验证
- 当前现状过于抽象
- 关键约束完全缺失
- 用户把问题定义、解决方案、实现细节混在一起
- 希望直接让 AI 开做，但上下文明显不足

## v0.2 轻量版流程

### 适用场景

适合：

- 新收到的模糊任务
- 还没正式立项的小问题
- 局部工作项
- 想先快速判断“这个问题值不值得继续展开”

### 轻量版步骤

只走 4 步：

1. 用一句话重述问题
2. 填最小必填字段 5 项
3. 判断是否通过门禁
4. 决定下一步：
   - 继续澄清
   - 进入结构化分解
   - 升级到完整版流程

### 轻量版输出格式

```markdown
## 当前判断
- 一句话定义：

## 最小必填字段
- 任务目标：
- 当前现状：
- 已知约束：
- 成功标准：
- 未知项：

## 门禁结论
- 是否通过：
- 缺口：

## 下一步
- 建议动作：
- 理由：
```

## 完整版何时触发

当出现以下情况时，从轻量版升级到完整版：

- 涉及多个角色
- 涉及项目背景或客户背景
- 任务准备进入方案设计
- 任务准备交给 AI 批量推进
- 用户已经明确提到 `skill / harness`

## v0.2 结构化分解入口条件

只有在门禁通过后，才进入结构化分解。  
并且建议加一个更明确的入口判断：

如果下面 3 个问题都能回答，就可以开始分解：

1. 这个任务真正要交付的最小结果是什么？
2. 当前最主要的限制条件是什么？
3. 哪一部分最适合先交给 AI？

如果答不出来，先不要拆。

## v0.2 建议新增的实操规则

### 规则 1：先轻后重

默认先走轻量版，必要时再升级到完整版。

### 规则 2：先过门禁，再做分解

不要把“结构化分解”当成澄清问题的替代品。

### 规则 3：先问题，再沉淀

不要因为用户提到 `skill / harness` 就直接开始设计。  
先确认问题和方法是否已经稳定。

## 建议的文档改动清单

如果进入 `v0.2` 实作，建议按这个顺序改：

1. 在 `SKILL.md` 里增加“轻量版流程”章节
2. 在 `SKILL.md` 里明确“最小必填字段”
3. 在 `SKILL.md` 里明确“通过 / 不通过条件”
4. 在 `README.md` 里补一个“快速开始：轻量版”
5. 在 `decomposition-patterns.md` 里加“什么时候先别拆”

## 建议的试点验证方式

`v0.2` 不建议一上来大改所有内容，建议先试：

1. 选 2 到 3 个真实任务
2. 每个任务先走轻量版
3. 看是否能更快判断：
   - 是否继续澄清
   - 是否进入分解
   - 是否升级到完整版
4. 收集使用反馈：
   - 哪些字段最有帮助
   - 哪些字段仍然太重
   - 门禁通过条件是否够清楚

## v0.2 预期结果

如果这轮迭代有效，应该能看到下面这些变化：

- 第一次使用的人更容易上手
- 工程师不容易被完整方法论吓退
- “现在到底能不能往下做”更清楚
- 结构化分解的入口更稳定
- 后续再下沉到 harness 时，规则基础更扎实

## 一句话总结

`v0.2` 的目标不是让 `engineering-problem-gate` 更复杂，而是让它更容易被真实工程师用起来。
