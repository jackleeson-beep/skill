# Engineering Problem Gate

## Purpose

本 skill 用于企业内统一的问题定义入口。  
目标不是直接给方案或代码，而是先把任务收敛成可执行、可校验、可流转的问题定义，再用结构化分解把任务拆到 AI 易于实现的颗粒度，并帮助工程师判断后续是否应该沉淀为 `skill`、`harness`，以及这样做是否合理。

## Language Support

- 当用户使用中文时，默认用中文执行这套流程并输出结果。
- When the user writes in English, apply the same workflow and respond in English.
- 当用户中英混合时，优先使用用户的主语言；必要时保留关键术语的中英文对应。
- 不因为语言不同而改变门禁标准、分解标准或验收标准。

默认假设：

- 这是项目流程中的**强制门禁**
- 模板字段必须填写
- 关键字段缺失时，不允许跳到下一阶段
- 所有角色都先走通用字段，再补角色扩展字段
- 输出不仅服务当前任务，也服务后续 `skill / harness` 的沉淀判断
- 问题清楚后，必须继续做结构化分解，不能把大任务整体丢给 AI

## Use This Skill When

当用户在讨论以下内容时，优先应用本 skill：

- 项目启动、需求进入、任务立项
- 新任务、模糊任务、需求澄清
- 方案前置讨论
- 实施前任务定义
- 验证前范围确认
- 交付前边界确认
- agent 工作流、harness、skill、门禁、流程规范

## Project Intake First

如果是一个新项目、新模块、新客户方向，或首次把本 skill 引入某个项目，先做项目背景采集，再进入具体任务门禁。  
项目背景采集模板见 [project-intake.md](project-intake.md)。

没有最基本的项目背景时，不要直接开始：

- 写方案
- 设计 skill / harness
- 拆实现任务
- 让 AI 直接执行

至少先拿到以下项目级上下文：

- 项目目标
- 产品/技术背景
- 目标客户或目标用户
- 当前阶段
- 可用资源
- 已有资料
- 关键约束
- 成功标准
- AI 当前可访问的材料范围

## Phase Model

所有任务都必须先判断当前所处阶段：

1. `需求/问题定义`
2. `方案设计`
3. `实现`
4. `验证`
5. `交付/支持`

先判断当前阶段，再决定本轮输出。  
如果阶段未明确，默认停留在 `需求/问题定义`。

## Gate Rules

严格执行以下规则：

1. 先填写模板，再给建议。
2. 模板缺项时，先追问缺口，不直接推进。
3. 不替用户脑补关键前提。
4. 不把“问题定义、解决方案、实现细节”混为一谈。
5. 只有在字段足够完整时，才允许建议进入下一阶段。
6. 如果用户想做 `skill` 或 `harness`，先判断是否真的值得沉淀，而不是默认创建。
7. 问题一旦明确，优先回到第一性原理，并把任务拆成更小、更稳定、更易交给 AI 的子问题。

## Mandatory Workflow

按顺序执行，不跳步：

### Step 0: 判断是否缺少项目背景

如果当前任务依赖项目背景，但项目背景尚未建立，先暂停具体任务分析，转为采集项目背景。

优先检查：

- 这是一次性小问题，还是项目内持续问题
- 是否已经知道项目目标和边界
- 是否已经拿到关键文档、案例、日志、规范、代码、知识库
- 是否清楚 AI 能访问哪些资源、不能访问哪些资源

如果上述信息明显不足，先走 `project-intake.md`，再继续后续步骤。

### Step 1: 一句话定义当前任务

输出一句话：

- 这个任务真正要解决什么问题
- 当前为什么需要处理它

### Step 2: 标记当前阶段

从五阶段模型中选一个当前阶段，并说明理由。

### Step 3: 填写通用字段

以下字段必须逐项填写；缺失就明确写 `未知` 或 `待确认`。

1. `任务目标`
2. `当前现状`
3. `已知约束`
4. `成功标准`
5. `未知项`
6. `影响范围`
7. `当前阶段`
8. `下一步允许动作`

### Step 4: 填写角色扩展字段

根据当前角色补充专属字段。  
角色字段清单见 [role-fields.md](role-fields.md)。

### Step 5: 做三层拆分

把当前内容拆成三层：

- `问题定义`
- `解决方案`
- `实现细节`

如果用户把三层混在一起，必须重新归类后再继续。

### Step 6: 判断门禁是否通过

只有同时满足下面条件，才算通过：

- 通用字段完整
- 角色字段完整
- 成功标准可检验
- 未知项已显式列出
- 当前阶段明确
- 下一步动作与当前阶段一致

### Step 7: 给出下一步

只允许输出一个主建议：

- 继续澄清
- 进入方案设计
- 进入实现
- 进入验证
- 进入交付/支持
- 沉淀为 skill / harness

如果未通过门禁，主建议只能是 `继续澄清`。

### Step 8: 做结构化分解

当门禁通过后，在进入实现或沉淀前，必须把任务拆成 AI 更容易完成的子问题。

拆分时使用以下原则：

- 每个子问题只解决一个核心目标
- 每个子问题都有清晰输入、输出和完成标准
- 每个子问题尽量减少隐含依赖
- 每个子问题应能由 AI 在单轮或少量轮次内稳定推进
- 如果一个子问题仍然过大，继续拆分

优先从第一性原理出发：

- 这个任务的本质目标是什么
- 不可绕过的约束是什么
- 哪些假设只是习惯，不是必需
- 哪个最小闭环能先证明方向正确

输出时至少给出：

- `问题本质`
- `最小子问题集合`
- `每个子问题的输入/输出`
- `哪些子问题适合直接交给 AI`

### Step 9: 如果涉及 skill / harness，做沉淀判断

当用户明确提到 `skill`、`harness`、流程固化、方法沉淀、规范化执行时，额外判断：

- 这是一次性上下文，还是稳定可复用模式
- 更适合做 `skill`、`harness`，还是两者组合
- 为什么这样设计是合理的
- 是否真的有助于加快项目推进，而不是增加流程负担

输出时要明确回答：

- `是否值得沉淀`
- `建议沉淀形态`
- `理由`

## Asking Rules

追问时遵循以下顺序：

1. 先补 `任务目标`
2. 再补 `成功标准`
3. 再补 `已知约束`
4. 再补角色字段
5. 最后确认 `下一步允许动作`
6. 如果用户提到 `skill / harness`，补问“为什么要沉淀、复用对象是谁、希望约束哪一层”

一次只追最关键的 1 到 3 个缺口。  
不要在字段未齐时直接开始设计实现方案。

当门禁已通过但问题仍然太大时，继续追问：

- 最小可交付结果是什么
- 哪一步最适合先让 AI 做
- 哪些依赖必须先固定

如果看起来“任务说得清，但项目背景不清”，优先追问：

- 这个任务属于哪个项目或客户方向
- 当前已有哪类项目资料
- 哪些历史案例或知识资产可复用
- 哪些资源 AI 当前可以使用

## Harness vs Skill Split

如果用户讨论流程落地，使用以下分工：

- `Harness`
  - 项目背景采集门禁
  - 强制模板
  - 强制回答
  - 阶段记录
  - 放行校验
  - 分解后再放行
  - 不通过时阻止越级
- `Skill`
  - 项目背景采集模板
  - 字段解释
  - 填写方法
  - 追问规则
  - 示例模板
  - 角色字段参考
  - 结构化分解方法
  - 第一性原理检查点

当目标是帮助工程师减少盲目性时，优先用本 skill 先证明“问题和方法已经稳定”；只有稳定后，再建议把规则下沉到 harness，或把方法沉淀成 skill。

## Full-Lifecycle Usage

本 skill 贯穿全流程使用，但每一阶段关注点不同：

- `项目启动`：先采集项目背景、资源、约束、成功标准、AI 可用资料
- `需求/问题定义`：明确目标、边界、约束、成功标准
- `方案设计`：明确问题本质、候选方案、取舍依据、接口边界，并做结构化分解
- `实现`：明确输入输出、依赖、完成条件
- `验证`：明确验证范围、通过标准、风险点
- `交付/支持`：明确交付物、影响范围、现场限制、回退方式

每次阶段切换，都应重新填一轮与当前阶段相关的字段，而不是只在最开始填写一次。  
如果某个模式在多个项目、多个角色、多个阶段中都重复出现，再考虑把它升级为正式 `skill` 或 `harness`。  
如果问题还没被拆到 AI 易实现的颗粒度，不要急着进入实现或沉淀。

## Example

### Input

“我们要让所有工程师用 Cursor 时先按统一模板把问题说清楚，不分任务类型，必须回答完才能继续。角色很多，包括数字、模拟、layout、软件、算法、测试、市场产品和现场支持。最终希望通过这个模式，让工程师做出合理的 skill 或 harness，少走弯路。问题清楚后，还要帮他们把任务拆成 AI 容易完成的子问题。”

### Output

```markdown
## 项目上下文
- 是否已完成项目背景采集：否
- 项目名称或代号：待确认
- 项目当前阶段：项目启动
- 可用资料与资源：待确认

## 当前判断
- 角色：项目流程设计者
- 当前阶段：需求/问题定义
- 一句话定义：需要建立一个面向全员工程师的 Cursor 问题定义门禁，要求统一模板、强制回答、贯穿项目全流程。

## 通用字段
- 任务目标：让工程师在进入方案或执行前，先完成结构化问题定义。
- 当前现状：全员配备 Cursor，但当前缺少统一入口，容易在问题未定义清楚时直接推进执行。
- 已知约束：
  - 不区分任务类型
  - 所有角色都要适用
  - 必须强制模板和强制回答
- 成功标准：所有角色都能使用统一骨架完成任务定义，并在字段不完整时被拦下。
- 未知项：
  - 各角色最小扩展字段集
  - 嵌入现有流程的具体节点
- 影响范围：公司内多角色工程师及其项目协作流程。
- 下一步允许动作：进入方案设计

## 角色扩展字段
- 目标岗位集合：数字、模拟、layout、软件、算法、测试、市场产品、现场支持
- 门禁强度：强制模板，强制回答
- 生命周期要求：贯穿需求、方案、实现、验证、交付

## 三层拆分
- 问题定义：如何建立统一的 Cursor 问题定义门禁
- 解决方案：通用字段加角色扩展字段的企业级 skill 与 harness 组合
- 实现细节：字段设计、校验逻辑、阶段切换规则

## 结构化分解
- 问题本质：如何让工程师在使用 Cursor 时先想清楚问题，再把任务拆到 AI 可稳定执行的颗粒度。
- 子问题 1：设计全员通用字段
- 子问题 2：设计角色扩展字段
- 子问题 3：设计通过/不通过门禁规则
- 适合直接交给 AI 的部分：模板生成、字段整理、角色字段草案、流程文档初稿

## 门禁结论
- 是否通过：通过
- 缺口：需要继续细化各角色扩展字段

## 下一步
- 建议动作：进入方案设计
- 理由：目标、范围、门禁强度和适用对象已经清楚，可以开始设计字段结构与流程挂点。

## 沉淀判断
- 是否值得沉淀：是
- 建议形态：skill + harness
- 理由：当前既需要统一的方法模板，也需要流程级强制门禁，单独使用 skill 或 harness 都不够完整。
```
