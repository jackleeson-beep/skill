# Decomposition Patterns

本文件为 `engineering-problem-gate` 提供结构化分解参考。  
用途是帮助工程师在问题已澄清后，把任务继续拆到 `AI 易于实现` 的颗粒度。

## 什么叫 AI 易实现颗粒度

一个子问题满足下面大部分条件时，通常就到了 AI 更容易稳定执行的颗粒度：

- 只解决一个核心目标
- 输入明确
- 输出明确
- 完成标准明确
- 依赖数量少
- 不需要同时跨越太多知识域
- 不要求 AI 一次完成长链路、多角色、多阶段协同
- 人可以在 1 到 3 句话内说清这个子问题在做什么

如果一个任务仍然包含：

- 多个目标
- 多种输出物
- 多个角色同时协作
- 过多前置假设
- 很多隐含依赖

说明它还不够小，应该继续拆分。

## 第一性原理检查表

在拆分前，先回答下面几个问题：

1. 这个任务真正要达成的结果是什么？
2. 哪些约束是不可绕过的？
3. 哪些做法只是习惯，不是本质要求？
4. 如果从零开始重建，最小闭环是什么？
5. 哪个最小验证动作最能证明方向对不对？

如果这些问题答不出来，先不要进入细分步骤。

## 标准拆分模板

每个子问题默认用这个格式表达：

```markdown
### 子问题 N
- 目标：
- 输入：
- 输出：
- 完成标准：
- 依赖：
- 是否适合直接交给 AI：
- 需要人工把关的点：
```

## 通用拆分顺序

大多数工程任务可以优先按这个顺序拆：

1. 先拆 `目标定义`
2. 再拆 `约束固定`
3. 再拆 `接口或边界`
4. 再拆 `核心执行单元`
5. 再拆 `验证与验收`

不要一上来就按“功能列表”拆。  
优先按“问题本质 -> 约束 -> 最小闭环 -> 验证”来拆。

## 常见错误拆分方式

### 错误 1：按组织结构拆

例如：

- 软件组做一部分
- 算法组做一部分
- 测试组做一部分

问题：这只是责任分配，不是 AI 易执行的任务分解。

### 错误 2：按大阶段粗拆

例如：

- 做方案
- 做实现
- 做验证

问题：颗粒太粗，AI 无法稳定完成。

### 错误 3：把未知项混进执行项

例如：

- 实现接口并确认接口到底该怎么定义

问题：一个子问题里混入了“定义问题”和“执行问题”。

## 适合直接交给 AI 的子问题类型

以下类型通常适合优先交给 AI：

- 模板生成
- 字段整理
- 规范比对
- 草案初稿
- 方案对比表
- 风险清单
- 用例枚举
- 文档重写
- 已知规则下的局部实现

以下类型通常需要人工先收敛，再交给 AI：

- 目标本身不清楚
- 约束彼此冲突
- 成功标准无法检验
- 跨多个角色的复杂协同
- 高风险设计取舍尚未对齐

## 多角色示例

### 数字工程师

原始问题：

“这个模块 timing 老过不了，想让 AI 帮忙处理一下。”

不好的拆法：

- 优化 timing

更好的拆法：

```markdown
### 子问题 1
- 目标：明确 timing 问题发生在哪个路径组
- 输入：时序报告、模块边界、约束文件
- 输出：问题路径分类结果
- 完成标准：能明确列出主要违例路径类型
- 依赖：最新 STA 报告
- 是否适合直接交给 AI：是
- 需要人工把关的点：报告是否为最新版本

### 子问题 2
- 目标：确认违例是约束问题、结构问题还是实现问题
- 输入：路径分类结果、约束文件、关键逻辑说明
- 输出：根因判断
- 完成标准：根因被归入单一主类
- 依赖：子问题 1 输出
- 是否适合直接交给 AI：部分适合
- 需要人工把关的点：主路径判断是否符合设计意图

### 子问题 3
- 目标：给出 2 到 3 个局部可执行优化动作
- 输入：根因判断
- 输出：可执行优化建议
- 完成标准：每个建议都对应明确对象和预期收益
- 依赖：子问题 2 输出
- 是否适合直接交给 AI：是
- 需要人工把关的点：优化动作对面积/功耗的副作用
```

### 模拟工程师

原始问题：

“这个指标在 PVT 下飘得厉害，想让 AI 帮分析。”

更好的拆法：

```markdown
### 子问题 1
- 目标：明确是哪个指标在什么 PVT 条件下异常
- 输入：仿真结果、目标指标、PVT 条件
- 输出：异常条件列表
- 完成标准：异常条件被结构化列出
- 依赖：完整仿真数据
- 是否适合直接交给 AI：是
- 需要人工把关的点：数据口径是否一致

### 子问题 2
- 目标：判断异常更可能来自电路拓扑、偏置还是版图寄生
- 输入：异常条件列表、电路说明、历史经验
- 输出：可能根因排序
- 完成标准：给出有依据的优先级
- 依赖：子问题 1 输出
- 是否适合直接交给 AI：部分适合
- 需要人工把关的点：根因推断是否符合电路常识
```

### 软件工程师

原始问题：

“帮我把这个平台功能补一下。”

更好的拆法：

```markdown
### 子问题 1
- 目标：定义功能的输入、输出和边界
- 输入：需求描述、现有接口
- 输出：功能边界说明
- 完成标准：输入输出与异常场景可列举
- 依赖：现有接口文档
- 是否适合直接交给 AI：是
- 需要人工把关的点：边界是否符合产品预期

### 子问题 2
- 目标：将功能实现拆成若干局部改动点
- 输入：功能边界说明、现有代码结构
- 输出：改动点列表
- 完成标准：每个改动点都可独立描述
- 依赖：代码结构信息
- 是否适合直接交给 AI：是
- 需要人工把关的点：拆分是否影响现有系统稳定性
```

### 算法工程师

原始问题：

“想优化一下效果，让方案更有竞争力。”

更好的拆法：

```markdown
### 子问题 1
- 目标：明确竞争力对应的评价指标
- 输入：当前指标、业务目标、对比基线
- 输出：核心指标定义
- 完成标准：能明确排序主目标和次目标
- 依赖：可用基线
- 是否适合直接交给 AI：是
- 需要人工把关的点：指标选择是否真实反映业务价值

### 子问题 2
- 目标：拆分哪些因素在限制当前指标
- 输入：实验记录、数据条件、模型结构
- 输出：限制因素列表
- 完成标准：每个因素都能关联到指标影响
- 依赖：实验可追溯数据
- 是否适合直接交给 AI：部分适合
- 需要人工把关的点：因果关系是否被过度推断
```

### 市场产品经理

原始问题：

“客户提了很多诉求，帮我整理一个方案。”

更好的拆法：

```markdown
### 子问题 1
- 目标：把客户诉求拆成场景、问题、期望结果
- 输入：客户反馈、会议纪要
- 输出：结构化需求列表
- 完成标准：每条诉求都有场景和目标
- 依赖：原始反馈资料
- 是否适合直接交给 AI：是
- 需要人工把关的点：是否遗漏关键信号

### 子问题 2
- 目标：区分真实需求、实现建议和情绪表达
- 输入：结构化需求列表
- 输出：分类结果
- 完成标准：每条诉求被归到单一主类
- 依赖：子问题 1 输出
- 是否适合直接交给 AI：是
- 需要人工把关的点：分类是否偏离客户原意
```

## 什么时候说明“还不能拆”

出现以下情况时，应先回到问题定义阶段，而不是硬拆：

- 目标不明确
- 成功标准缺失
- 约束冲突
- 关键输入缺失
- 角色都还没确定

这时应明确说：

`当前不适合继续分解，先补齐问题定义。`

## 使用建议

1. 先判断问题是否已经足够清楚。
2. 再做第一性原理检查。
3. 再按标准模板拆分子问题。
4. 优先找出最小可交付闭环。
5. 最后才决定哪些部分交给 AI、哪些部分保留人工判断。
