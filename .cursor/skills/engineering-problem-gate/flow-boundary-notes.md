# Engineering Problem Gate 边界说明

本文件用于记录 `engineering-problem-gate` 在真实测试中的适用边界。  
目标不是继续扩展能力，而是防止这套流程被用得过重、过深、过宽。

## 一句话边界

`engineering-problem-gate` 的职责，是把任务送到“可设计、可拆分、可交给 AI”的状态。  
它不是最终方案设计器，也不是完整项目实施框架。

## 这套流程负责什么

它主要负责：

1. 补齐项目背景与资源
2. 定义当前问题
3. 收敛目标、现状、约束、成功标准、未知项
4. 判断是否通过门禁
5. 把问题拆到 AI 更容易执行的颗粒度
6. 判断是否值得沉淀为 `skill / harness`

## 这套流程不负责什么

它不应该直接承担：

1. 完整的大型系统架构设计
2. 详细的项目实施计划
3. 跨阶段的长期路线图设计
4. 最终技术方案拍板
5. 完整 harness 的替代品

## 轻量版的边界

`v0.2` 轻量版最适合处理：

- 模糊但还不复杂的任务
- 还没正式进入设计阶段的新任务
- 想先判断“值不值得继续往下做”的问题
- 想先判断“能不能进入结构化分解”的问题

### 轻量版能做到

- 快速重述问题
- 填最小必填字段
- 判断通过 / 不通过
- 决定下一步动作

### 轻量版不能替代

- 正式设计文档
- 完整架构设计
- 复杂项目路线图

### 轻量版的合理终点

轻量版最合理的终点通常只有三个：

1. `继续澄清`
2. `进入结构化分解`
3. `升级到完整版流程`

如果任务已经进入大型架构设计，就不应继续停留在轻量版。

## 完整版的边界

完整版适合处理：

- 跨角色任务
- 跨模块任务
- 项目级任务
- 准备进入正式方案设计的任务
- 准备进一步沉淀为 `skill / harness` 的任务

### 完整版负责把任务整理成

- 可设计
- 可讨论
- 可拆分
- 可进入后续实施流程

### 完整版仍然不负责

- 替代最终方案评审
- 替代详细实现设计
- 替代完整项目管理

## 什么时候该升级

从轻量版升级到完整版，通常有以下信号：

- 任务涉及多个角色
- 任务依赖项目背景
- 任务开始进入方案设计
- 任务准备交给 AI 批量推进
- 用户已经在讨论 `skill / harness`
- 问题已经清楚，但拆分结果会影响架构边界

## 什么时候不该继续硬推 gate

如果已经出现下面情况，就不要继续用 `gate` 去硬撑：

- 问题已经非常清楚，且只是局部实现任务
- 已经进入明确的方案设计与评审阶段
- 需要的是详细技术方案，而不是问题门禁
- 需要的是项目管理，而不是问题收敛

这时应该把任务交给：

- 方案设计流程
- 架构评审流程
- 实施计划流程
- 或后续 harness / skill 专项设计流程

## 本次测试验证出的结论

通过 SDK 重构案例，可以确认：

1. 轻量版适合把“大方向”推进到“是否可以正式设计”的节点
2. 最小必填字段足以拦住“还没想清楚就开做”
3. 成功标准一旦量化，任务就能顺利通过门禁
4. 一旦任务进入大型系统重构，就必须升级到完整版
5. `gate` 最适合作为统一入口，而不是最终方案承载层

## 推荐的流程边界

建议把这套内容理解成以下链路：

1. `project-intake`
   - 解决项目背景是否完整
2. `gate`
   - 解决当前问题是否清楚、是否放行
3. `decomposition`
   - 解决是否拆到 AI 易执行
4. `design / implementation / harness`
   - 进入后续专项流程

## 一句话总结

`engineering-problem-gate` 的最佳定位，是“前置门禁和统一入口”，而不是“把所有后续设计工作都吸进来”。
