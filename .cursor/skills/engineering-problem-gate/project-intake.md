# Project Intake

本文件用于在项目启动前采集背景信息。  
目标是让 `engineering-problem-gate` 不只处理单条问题，还能和真实项目结合。

没有完成这份采集前，不建议直接：

- 写详细方案
- 创建正式 skill
- 创建正式 harness
- 让 AI 大规模执行项目任务

## 采集目标

先让 AI 明白：

- 这个项目到底在做什么
- 为什么现在做
- 对谁有价值
- 已经有哪些资源
- 缺什么关键材料
- AI 可以访问什么，不能访问什么

## 最小必填字段

如果时间很紧，至少先补齐这些：

- 项目目标
- 当前痛点
- 目标用户
- 当前资源
- 关键约束
- 成功标准
- AI 可访问范围

缺少这些时，AI 很容易做出“看起来合理、实际脱离项目”的输出。

## 常见问题

### 问题 1：只有一个模糊方向，没有正式项目资料

做法：

- 先把方向写成 1 句话
- 列出已知客户或场景
- 列出当前最容易拿到的资料
- 明确哪些是假设，哪些是事实

### 问题 2：资料很多，但 AI 不知道从哪开始

做法：

- 不要一次丢所有资料
- 先列“最关键的 5 类资源”
- 标注每类资源解决什么问题
- 标注哪些资源当前最可信

### 问题 3：不同角色理解的项目边界不一样

做法：

- 先写统一的项目目标
- 再写每个角色各自关心的点
- 明确哪些是共享上下文，哪些是角色视角差异

## 资源整理建议

项目资源建议按下面方式归类：

1. `目标与需求`
2. `客户与场景`
3. `技术资料`
4. `历史案例`
5. `现有工程资产`
6. `验证与交付资料`

这样后续在做问题定义、结构化分解、skill / harness 设计时，AI 更容易找到对应材料。

## 与主 Skill 的关系

- `project-intake.md`：解决“项目背景是否完整”
- `SKILL.md`：解决“当前任务是否定义清楚、能否放行”
- `decomposition-patterns.md`：解决“问题清楚后，如何拆到 AI 易执行”
- `role-fields.md`：解决“不同角色还要补哪些字段”

## 使用建议

1. 新项目第一次使用时，先填写本文件。
2. 项目背景发生明显变化时，更新本文件。
3. 每次进入重大阶段前，检查这里的内容是否仍然有效。
4. 如果后续要做企业内正式 harness，可把这里的最小必填字段做成启动门禁。
