# Engineering Problem Gate v0.3 迭代草案

## 版本定位

`v0.3` 不是继续给 `engineering-problem-gate` 增加更多字段。  
它的目标是完成一次**定位升级**：

- 从“问题门禁 skill”
- 升级为“交付入口 skill”

也就是说，`gate` 仍然负责开门和放行，但它不再只是为了判断问题是否清楚，而是为了把任务送进完整的交付链路。

## 为什么要升级到 v0.3

经过前面的测试，可以看到：

- `v0.1` 解决了基础框架问题
- `v0.2` 解决了首次使用门槛和轻量版流程
- 但仍然缺少一个上层定位：
  - 开门前做什么
  - 进门后做什么
  - 什么时候启动 AutoResearch
  - 什么时候必须人审

因此，`v0.3` 要回答的不是“再加哪些字段”，而是：

> `engineering-problem-gate` 在整个 AI 交付链路里，到底扮演什么角色？

## v0.3 核心目标

本轮只做四件事：

1. 明确 `gate` 是完整交付链路的入口
2. 把“第一性原理看透问题”前置
3. 把“结构化分解”定义为门禁通过后的下一步
4. 把“闭环空间判断 + AutoResearch 触发”纳入后续链路

## v0.3 设计原则

1. **不把所有内容塞进 gate**
   - `gate` 负责入口
   - 后续交付链路由上层框架承接

2. **问题看透要发生在开门前**
   - 如果问题本质还没看透，不要让任务通过门禁

3. **结构化分解是 gate 的终点，不是后续设计的替代品**
   - `gate` 负责把任务送到“可分解”的状态

4. **AutoResearch 只在闭环空间中启动**
   - 不是所有问题都适合直接启动多轮迭代

5. **human-in-the-loop 必须保留**
   - gate 不能假装替代人工验收

## v0.2 到 v0.3 的主要升级点

### 升级点 1：前置第一性原理

在 `v0.2` 中，第一性原理已经存在，但更像分析方法。  
在 `v0.3` 中，它应该变成**开门前的前置要求**。

也就是说，至少要能回答：

- 问题本质是什么
- 不可绕过的约束是什么
- 最小正确方向是什么

否则就不能顺利放行。

### 升级点 2：gate 的终点更清晰

`v0.2` 中，gate 的终点是：

- 继续澄清
- 进入结构化分解
- 升级到完整版

`v0.3` 中，应进一步明确：

- `gate` 的目标是把任务送到“可设计、可拆分、可交给 AI”的状态
- 一旦进入完整架构设计，就不应继续把 `gate` 当成主承载层

### 升级点 3：加入闭环空间判断

不是所有经过结构化分解的任务都能直接启动 AutoResearch。  
`v0.3` 应明确增加：

- 什么叫闭环空间
- 哪些子问题已具备闭环条件
- 哪些子问题还应该继续收敛

### 升级点 4：human-in-the-loop 明确化

`v0.3` 应明确写出：

- 哪些阶段必须人工确认
- 哪些结果只能作为 AI 候选输出
- 什么情况下必须由人做最终裁决

## v0.3 的角色定位

在完整链路中，`engineering-problem-gate` 的定位应明确为：

- `See Through` 的主执行入口
- `Structure` 的启动入口
- `Close the Loop` 的前置判断器
- 不是 `Accept and Deliver` 的替代品

## v0.3 建议新增的规则

### 规则 1：开门前先看透

不能回答“问题本质”和“最小正确方向”时，不放行。

### 规则 2：分解是放行后的默认下一步

通过门禁后，默认先做分解，而不是直接进入实现。

### 规则 3：闭环前不启动 AutoResearch

如果子问题还不满足闭环条件，不要过早进入多轮迭代。

### 规则 4：人审点明确

在以下节点必须人工确认：

- 问题本质是否成立
- 结构化分解是否合理
- 子问题是否进入闭环空间
- 输出是否达到交付要求

## 建议的文档改动清单

如果进入 `v0.3` 实作，建议按这个顺序改：

1. 在 `SKILL.md` 中加入“第一性原理前置判断”
2. 在 `SKILL.md` 中加入“闭环空间判断”
3. 在 `SKILL.md` 中增加明确的人审点
4. 在 `README.md` 中说明 `gate` 是完整交付链路的入口
5. 用 `ai-delivery-loop.md` 作为上层框架说明

## 建议的试点验证方式

本轮不建议先改很多文字，而建议先验证定位：

1. 拿 2 个真实任务
2. 看“第一性原理前置判断”是否有用
3. 看 gate 是否能稳定把任务送到“可分解”状态
4. 看闭环空间判断是否能拦住过早启动 AutoResearch

## v0.3 预期结果

如果这轮升级有效，应该能看到：

- `gate` 的职责边界更清晰
- 团队不再把 `gate` 当成万能流程
- 第一性原理判断真正前置
- 结构化分解和 AutoResearch 启动条件更清楚
- 后续设计 harness 时，阶段边界更容易落地

## 一句话总结

`v0.3` 的核心不是“加更多内容”，而是让 `engineering-problem-gate` 从问题门禁，升级成完整交付链路的统一入口。
