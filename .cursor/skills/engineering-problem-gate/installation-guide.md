# Engineering Problem Gate 打包与安装指南

## 推荐推广方式

公司内推广建议按下面优先级：

1. **项目级共享安装**
   - 适合团队协作
   - 所有人在同一项目里看到相同 skill
   - 推荐作为默认方式

2. **个人级安装**
   - 适合先试点、先小范围验证
   - 个别工程师可以先在自己环境中试用

3. **压缩包分发**
   - 适合通过邮件或内部知识库快速扩散
   - 适合作为补充方式，不建议长期依赖

## 推荐目录结构

建议完整打包下面这些文件：

```text
engineering-problem-gate/
├── SKILL.md
├── role-fields.md
├── decomposition-patterns.md
├── project-intake.md
├── rollout-intro.md
├── installation-guide.md
└── email-template.md
```

## 方式一：项目级共享安装

适合正式推广。

### 安装位置

将整个目录放到项目仓库：

```text
.cursor/skills/engineering-problem-gate/
```

### 使用方式

工程师打开该项目后，Cursor 就可以在该项目上下文中发现并使用这个 skill。

### 优点

- 团队统一版本
- 与项目一起演进
- 便于做项目级规范
- 新成员进入项目后可直接使用

### 适合场景

- 公司内重点项目
- 跨角色协作项目
- 希望把 AI 使用方式纳入正式流程的团队

## 方式二：个人级安装

适合试点用户先体验。

### 安装位置

把整个目录复制到：

```text
~/.cursor/skills/engineering-problem-gate/
```

### 优点

- 上手最快
- 不依赖项目仓库改动
- 适合小范围先验证

### 缺点

- 版本不易统一
- 团队升级时需要逐人同步

## 方式三：压缩包分发

适合通过邮件、知识库、IM 群快速发放。

### 打包方式

把 `engineering-problem-gate` 整个目录打成一个 zip 包，例如：

```text
engineering-problem-gate-v1.zip
```

### 分发建议

- 邮件附件
- 公司内部网盘
- Confluence / Wiki / 文档中心
- IM 群公告

### 安装说明

收到压缩包后，用户只需要：

1. 解压
2. 复制到项目目录 `.cursor/skills/`  
   或复制到个人目录 `~/.cursor/skills/`

## 最低摩擦使用法

为了让大家更容易上手，建议推广时只要求他们先记住一句话：

> 在开始做方案、写代码、写 skill、写 harness 之前，先让 Cursor 用 `engineering-problem-gate` 帮你收敛问题。

建议附上一个最短示例：

```text
请先用 engineering-problem-gate 帮我梳理这个任务，不要直接进入方案或实现。
```

## 公司内推荐落地方式

建议按三步走：

### 第一步：小范围试点

- 先选产品经理、FAE、软件工程师各 1 到 2 人
- 连续试用 1 到 2 周
- 收集团队最常见问题和卡点

### 第二步：项目级推广

- 放进重点项目的 `.cursor/skills/`
- 在项目启动会或组会中统一介绍
- 要求大家在模糊任务上优先使用

### 第三步：流程级固化

- 将 `project-intake.md` 和 `SKILL.md` 中的关键字段收敛成公司模板
- 再考虑把其中稳定规则下沉到 `harness`

## 常见问题

### 1. 为什么不直接只发一个 `SKILL.md`？

可以，但不推荐。  
因为这套 skill 的价值不只在主流程，还在：

- 角色字段
- 项目背景采集
- 结构化分解模式
- 推广说明

如果只发主文件，执行质量会明显下降。

### 2. 如何让大家更容易接受？

不要把它包装成“多一道流程”，而要包装成：

- 少走弯路
- 少让 AI 跑偏
- 提高首次输出质量
- 帮大家更快得到可执行结果

### 3. 如何做版本管理？

建议在目录名不变的情况下，通过文档内容维护版本，并在邮件或知识库中标注：

- 当前版本号
- 更新时间
- 主要变化

如果后续正式化，也可以再加一个 `CHANGELOG.md`。
